# NTFS Test Corpus Documentation

This document describes the corpus of images generated by the `test-corpus/generate_test_corpus.sh` script. This corpus serves as the baseline for Continuous Integration (CI) validation to ensure that the repair engine behaves strictly according to the Clean Room Specification.

## Corpus Objective

The disk images (default 50 MB) simulate low-level corruptions on critical NTFS structures. Each image has a **strict** and **deterministic** expected outcome. If the engine produces a different outcome or crashes, the build must fail.

## Test Scenarios

### T-01: Baseline (Clean)
- **File:** `T-01-Clean.img`
- **State:** Freshly formatted image by `mkntfs`, without corruption.
- **Expected Outcome:** The engine traverses all 16 phases without raising any errors. No disk writes occur (even in `--apply` mode).

### T-02: MFT Record Corruption (USA Mismatch)
- **File:** `T-02-MFT-Corrupt.img`
- **Injection Method:** Writing `0xFFFF` at the Update Sequence Number (USN) offset at the end of the first MFT record sector.
- **Expected Outcome:** 
  1. The parser raises an `ERR_USA_MISMATCH` error when reading the record.
  2. The engine consults the Decision Matrix (Phase 4).
  3. The engine retrieves the intact record from the `$MFTMirr` (because the record is within the first 4 system records).
  4. The record is successfully restored.

### T-03: $MFTMirr Desynchronization
- **File:** `T-03-MFTMirr-Corrupt.img`
- **Injection Method:** Filling the Data Run hosting the `$MFTMirr` with random data (`/dev/urandom`).
- **Expected Outcome:** 
  1. The engine detects that the mirror has lost its `FILE` magic number.
  2. During Phase 15 (Final Sync), the "Semantic Validation Barrier" verifies that the primary `$MFT` is healthy.
  3. The primary `$MFT` overwrites the defective mirror.

### T-04: Bitmap Cluster Leaks
- **File:** `T-04-Bitmap-Corrupt.img`
- **Injection Method:** Overwriting the physical clusters containing the `$Bitmap`, falsely declaring clusters as allocated.
- **Expected Outcome:** 
  1. The "Double-Pass Rebuild" of Phase 14 generates a Ground Truth from valid Data Runs.
  2. The bit-by-bit reconciliation detects that `Computed = FREE` but `Disk = USED`.
  3. The tool clears the erroneous bits on disk after validating a consensus (Massive Release Barrier) and writing the WAL snapshot.

### T-09: Boot Sector Corruption
- **File:** `T-09-Boot-Corrupt.img`
- **Injection Method:** Zero-filling (`/dev/zero`) the first physical sector (LBA 0).
- **Expected Outcome:** 
  1. Phase 0: The primary Boot Sector lacks the `0x55AA` signature and the `NTFS` magic.
  2. The engine seeks to the last sector of the partition (Backup Boot Sector).
  3. The Backup overwrites the Primary. Mounting continues normally.

## Adding New Test Cases

To add a new scenario (e.g., `T-06-BTree-Corrupt`), modify the `generate_test_corpus.sh` script using `ntfsinfo` to find the target's LCN (`$INDEX_ALLOCATION`), then use `dd` with a byte offset (`LCN * cluster_size`) to inject a corrupted payload.
